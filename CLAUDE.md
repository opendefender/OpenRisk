# OpenRisk — Contexte permanent Claude Code
 
## Projet
OpenRisk est une plateforme GRC enterprise open-source (https://github.com/opendefender/OpenRisk).
Partie de l'écosystème OpenDefender. Marché cible : France, Belgique, Maghreb, Afrique subsaharienne.
 
## Stack technique
- Backend  : Go 1.25 · Fiber v2 · GORM · PostgreSQL 16 · Redis · golang-migrate
- Frontend : React 19 · TypeScript strict · Zustand · Tailwind CSS 3 · Recharts · React Router v7
- Infra    : Docker multi-stage · Kubernetes + Helm · GitHub Actions CI/CD
- Obs.     : Prometheus · Grafana · Loki · zerolog JSON · Sentry
## Architecture obligatoire
Backend — Clean Architecture stricte :
  /cmd/server/main.go         → DI container, graceful shutdown
  /internal/domain/           → entités pures (pas de Fiber, pas de GORM ici)
  /internal/application/      → use cases (1 use case = 1 fichier)
  /internal/infrastructure/   → repositories, messaging, integrations
  /internal/api/http/         → handlers Fiber + middleware
  /pkg/                       → packages partagés (scoring, cti, notify, export, ai, crq)
 
Frontend — Feature-based :
  /src/features/[module]/     → pages, components, hooks, stores par feature
  /src/shared/                → design system, hooks globaux, utils
  /src/services/              → client API typé (généré depuis OpenAPI)
  /src/locales/               → fr.json, en.json

10. Optimistic updates sur toutes les mutations critiques (UX perçue)
11. Zod validation côté client sur tous les formulaires
## Formule Score Engine
Score = Probability (0.0–1.0) × Impact (0.0–10.0) × AssetCriticality (0.1–3.0)
Criticality : score ≥ 7.0 = critical · ≥ 4.0 = high · ≥ 2.0 = medium · < 2.0 = low

## Smart Risk Calculation (spec §8 « Calcul de risque intelligent »)
Moteur **additionnel** au Score Engine classique (qui reste la source de vérité `Risk.Score`). Ne le remplace PAS.
- **Moteur pur** : `pkg/scoring/smart.go` (`ComputeSmart(SmartInput, FactorWeights) SmartResult`), stdlib, déterministe, testé. Le Score Engine `P×I×AC` (`engine.go`) est inchangé.
- **Formule** : `SmartScore = 100 × Σ (poids_normalisé_i × facteur_i)` sur **8 facteurs**, chacun normalisé dans `[0,1]` (1 = risque maximal). Bandes : ≥75 critical · ≥50 high · ≥25 medium · <25 low.
- **8 facteurs** (constantes `FactorKey`) et leur normalisation :
  1. `business_criticality` = `AssetCriticality.ScoreFactor()/3.0` (fallback impact/10×3 si pas d'actif) ;
  2. `internet_exposure` = tags risque (`internet-facing`→1.0, `internal`→0.1) sinon type d'actif ;
  3. `vulnerabilities` = `0.7×(maxCVSS/10) + 0.3×log-échelle(nb)` sur les vulns ouvertes de l'actif ;
  4. `control_maturity` = `1 − couverture_contrôles_implémentés` (neutre 0.5 si pas de compliance) ;
  5. `incident_history` = log-échelle du nb d'incidents passés de l'actif ;
  6. `exploitability` = max(EPSS, KEV→0.95, exploit→0.70, maturité) ;
  7. `financial_value` = `min(ALE_XAF / référence, 1)` (référence défaut 50 M FCFA, ALE via `pkg/crq`) ;
  8. `threat_intel` = signal CTI (KEV→1.0, sinon origine `cti_auto`/CVE→0.7).
- **Pondérations configurables par tenant** : `domain.RiskScoringWeights` (1 ligne/tenant, index unique `tenant_id`, 8 colonnes `numeric(6,4)`, défauts biz .15/expo .10/vuln .20/ctrl .10/incid .10/exploit .15/fin .10/cti .10). **Valeurs relatives** → `normaliseWeights` les ramène à somme 1.0 (clé absente = défaut, négatif = 0). `ToFactorWeights()`/`ApplyFactorWeights()` font le pont domaine↔moteur ; `Validate()` (chaque ∈ [0,1], au moins un > 0).
- **Persistance** : colonnes `smart_score`/`smart_level`/`smart_factors`(jsonb breakdown gelé)/`smart_computed_at` sur `risks` (migration `0035`, AutoMigrate `RiskScoringWeights` + colonnes). Write-back ciblé `GormRiskRepository.UpdateSmartScore` (pas de spam d'historique, comme `UpdateScore`).
- **Assemblage** : `application/risk/compute_smart_score.go` — `ComputeSmartScoreUseCase` (ports **optionnels** nil-safe via `With*` : assetRepo, vulnLister, compliance, incidents, quantifier, persister) ; `Execute(persist)` calcule+cache, `Preview(weights)` simule sans persister. Chaque source manquante dégrade son facteur, jamais d'échec.
- **Endpoints** : `GET /risks/:id/smart-score` (calcul+cache, `risks:read`), `POST /risks/:id/smart-score/simulate` (aperçu non-persisté, `risks:read`), `GET /risk-scoring/weights` (`risks:read`), `PUT /risk-scoring/weights` (**admin**). OpenAPI tag « Smart Risk Calculation ».
- **Frontend** : `features/risks/SmartRiskRadar.tsx` (radar Recharts + breakdown classé), onglet « Score intelligent » du drawer (`RiskRegisterPage`), écran `RiskWeightsSettings` (`/risks/weighting`, sliders + part effective normalisée, garde admin). Service/hooks typés `smartScoreService.ts`/`useSmartScore.ts`.

## Security Automation / SOAR (spec §10 « Automatisation »)
Moteur d'orchestration event-driven qui chaîne des actions de sécurité sans intervention humaine. **Additionnel** — réutilise les briques existantes (bus Redis, workers, `pkg/notify`, `pkg/ticketing`, `notification.UseCase`) sans les modifier.
- **Modèle** (`domain/automation.go`, migration `0036`, AutoMigrate) : `AutomationRule` (tenant-scoped : `trigger` + `conditions` jsonb + `actions` jsonb **ordonnées** + `sla` jsonb + `priority`) ; `AutomationExecution` (journal d'audit d'un déclenchement, `steps` jsonb, statut success/partial/failed/skipped) ; `SLATracker` (compte à rebours : `due_at`/`escalate_at`/`status` open→breached→escalated→met/closed) ; `AutomationChannelConfig` (1/tenant, Slack/Teams webhooks **write-only** + email de repli).
- **Moteur d'événements** : `application/automation.Engine.HandleTrigger(trigger, TriggerContext)` charge les règles **activées** du tenant pour ce trigger (`ListEnabledByTrigger`, ordre `priority`), évalue `matchConditions` (gates severity/CVSS/KEV/tier/tags — un zéro-value matche tout), puis exécute la chaîne. Chaque action = un `ExecutionStep`. **Ports d'action tous optionnels/nil-safe** (`WithNotifier/WithTicketer/WithRiskCreator/WithRiskAssigner/WithRiskResolver/WithAssetScanner`) : capacité absente → step « skipped », jamais d'échec dur. Le `TriggerContext` s'enrichit au fil de la chaîne (`create_risk` pose `RiskID` → `assign`/`ticket`/`start_sla` l'utilisent).
- **Triggers → cycle de vie des tâches d'arrière-plan** : l'ingest de vulnérabilités publie `events.VulnerabilityDetected` (canal Redis) sur une vuln **nouvellement créée** → `workers.AutomationWorker` (abonné Redis, comme `ScoreWorker`) décode l'event → `Engine.HandleTrigger`. Trigger `risk_score_updated` câblé de même sur `events.RiskScoreUpdated`. `workers.SLAMonitor` (ticker **1 min**, cross-tenant, comme `ScanScheduler`) fait 2 balayages par tick : `SweepEscalations` (escalade) + `SweepAutoClose` (clôture auto). Tous démarrés en goroutine dans `main.go` (bloc « 5.9 »).
- **Configurer un nouveau canal d'alerte** : (a) **côté tenant** — `PUT /automation/channels` pose les webhooks Slack/Teams + email de repli (`AutomationChannelConfig`, admin) ; le dispatcher `infrastructure/automation.Notifier` résout les destinataires (owner + rôle via `OrganizationMember`) et poste sur chaque canal (best-effort, renvoie les canaux livrés). (b) **ajouter un TYPE de canal** — implémenter le POST webhook dans `pkg/notify/chatops.go` (modèle `PostSlack`/`PostTeams`, sans dépendance, HTTPDoer testable), ajouter la constante `NotificationChannel`, brancher le `case` dans `Notifier.Notify`. Slack = format attachment, Teams = MessageCard.
- **Calcul du dépassement SLA + escalade** : `start_sla` crée un `SLATracker` avec `due_at = now + AutomationSLAConfig.MinutesFor(severity)` (repli sur le tier supérieur si 0) et `escalate_at = due_at + escalate_after_minutes`. Le `SLAMonitor` : `ListBreaching(now)` = trackers ouverts dont `escalate_at ≤ now` → escalade (notifie `escalate_to_role` sur `escalate_channels`, `status=escalated`, `escalation_level++`, repousse `escalate_at` d'1 h pour ré-escalader en cadence). `RemainingMinutes(now)` (négatif = dépassé) + `IsBreached` alimentent le dashboard. **Clôture auto (étapes 7-8 de la spec)** : `SweepAutoClose` balaie les trackers liés à un risque (`ListOpenLinkedToRisk`), interroge `RiskStateLookup.IsRiskResolved` (statut mitigated/closed/accepted) → `status=met` (dans les délais) ou `closed` (après dépassement).
- **Endpoints** : `/automation/rules` (CRUD + `POST /:id/test` dry-run qui bypasse enabled/trigger), `/automation/executions` (+ `/rules/:id/executions`), `/automation/sla` (+ `/stats`), `/automation/channels` (RBAC `automation:read`/`automation:write`, écriture canaux admin). Sous-chemins statiques (`executions`/`sla`/`channels`) montés **avant** `/:id` (piège Fiber).
- **Frontend** `features/automation` : `AutomationPage` à 4 onglets (deep-link `?tab=`) — **Règles** (constructeur de workflow : chaque règle rend sa chaîne trigger→actions en chips colorées ; toggle/test/edit/delete), **SLA en cours** (KPI open/at-risk/breached/escalated/met + table avec barre de budget consommé + temps restant/dépassé, refetch 30 s), **Historique** (exécutions dépliables par step), **Canaux** (config Slack/Teams/email) ; `RuleEditorModal` (trigger + conditions + chaîne d'actions add/reorder/remove avec params par action + politique SLA). Service/hooks typés `automationService.ts`/`useAutomation.ts` (zéro `any`).

## IA — Assistant GRC (spec §12 « IA »)
Fonctionnalités d'IA qui assistent les équipes GRC. **Additionnel** — généralise le connecteur LLM du Board Report (`pkg/ai.Advisor`) sans le modifier ; **second client LLM du projet**.
- **Service unifié pur** : `pkg/ai.Assistant` (interface + DTOs **purs**, zéro import `internal/`/GORM, donc testable trivialement). Deux implémentations : **`ClaudeAssistant`** (SDK officiel `anthropics/anthropic-sdk-go`, modèle `claude-opus-4-8`, **adaptive thinking**, prompts système imposant un **objet JSON strict** décodé par méthode) ; **`TemplateAssistant`** (déterministe, sans réseau ni clé — le repli). - **Stratégie de prompt-engineering** : un prompt **système** par capacité fixe le rôle (expert GRC/analyste menaces/auditeur), la langue (`locale` FR/EN), et **exige un objet JSON valide** avec des clés nommées (pas de Markdown) ; le prompt **utilisateur** injecte le contexte agrégé en JSON compact « chiffres à ne pas modifier ». `parseInto` tolère un fence ```json``` ou du texte autour (extraction du 1er `{` au dernier `}`). Ports **tous optionnels/nil-safe** (`WithRisks/WithCompliance/WithVulns/WithOrgs`) : une source absente dégrade sa tranche, jamais d'échec. ## Gouvernance (spec §15 « Gouvernance »)
Système de gouvernance à 4 piliers. **Additionnel** — le RBAC existant (`domain/rbac.go` : rôles custom + permissions granulaires) est conservé ; on ajoute la piste d'audit immuable, les délégations temporelles et le moteur d'approbation Maker-Checker. Tout est tenant-scoped (RÈGLE #2). Branche `feature/governance-audit-workflows`.
- **Domaine** : `internal/domain/governance.go` — `AuditEvent` (append-only, pas d'`UpdatedAt`/soft-delete), `Delegation` (`IsActiveAt(t)`), `ApprovalWorkflow` (config : chaîne d'étapes `WorkflowStep` liée à `(entity_type, action)`) + `ApprovalRequest` (state machine : `Steps` snapshotés, `Decisions` jsonb, `CurrentStep`). Marqueur `Auditable` + opt-ins dans `governance_auditable.go`. Types jsonb (`JSONMap`, `WorkflowStepList`, `ApprovalDecisionList`) avec `Valuer/Scanner` comme l'automation. Migration `0037` + AutoMigrate.
- **⭐ Piste d'audit — « comment garantir que les développeurs n'oublient pas de logguer »** : un **plugin GORM** (`internal/infrastructure/audittrail/plugin.go`, installé dans `main.go` via `database.DB.Use(audittrailinfra.New(database.DB))`) enregistre des callbacks `after_create`/`before_update`/`after_update`/`after_delete`. **Tout modèle qui implémente `domain.Auditable` (méthode `AuditEntityType() string`, 1 ligne) est journalisé automatiquement** à chaque mutation struct-form — aucun appel manuel requis. **Pour couvrir une nouvelle entité : ajouter la ligne `func (X) AuditEntityType() string { return "x" }` dans `governance_auditable.go`.** Le snapshot Avant→Après est pris par **json-marshal** du record → les champs `json:"-"` (mots de passe, secrets) ne sont **jamais** capturés (RÈGLE #6). Le plugin est **strictement best-effort** (recover sur panic, erreurs avalées) : un échec d'audit ne casse **jamais** l'écriture métier. Il **saute** les mutations sans `tenant_id` (jamais de fuite cross-tenant) et la table `audit_events` elle-même. **L'acteur (« Qui »)** : un middleware monté sur le groupe `protected` (dans `main.go`) estampille `Actor{ID, TenantID, IP, UA}` sur `c.UserContext()` via `audittrail.WithActor` ; tout repo qui fait `db.WithContext(c.UserContext())` (asset, compliance…) propage l'acteur au plugin. **Limite honnête** : les `Updates(map[string]interface{})` (ex. update d'Asset) sont journalisés sans diff (le PK n'est pas dans le reflect map) → pour un diff garanti, utiliser le `Recorder` explicite. **`Recorder`** (`application/governance/audit_usecases.go`) = écriture explicite best-effort/nil-safe pour les actions à forte valeur (approbations, délégations, export) ; les use cases l'appellent. L'ancien `AdminAuditEvent` (jamais câblé) est **supplanté**.
- **Moteur d'approbation (structure)** : `application/governance/approval_usecases.go`. Un admin crée un `ApprovalWorkflow` liant `(entity_type, action)` (ex. `risk_acceptance`/`accept`) à une chaîne ordonnée d'étapes (chacune : `approver_role` éligible + `min_approvals`). `SubmitApprovalRequest` trouve le workflow (`FindWorkflow`), **snapshote ses étapes** sur la requête (les éditions ultérieures du workflow ne réécrivent pas les requêtes en vol) → `pending`, step 0. **`DecideApprovalStep` = le cœur du contrôle Maker-Checker** : impose (1) **four-eyes** — le demandeur (`RequestedBy`) ne peut jamais approuver sa propre requête ; (2) **éligibilité par rôle** par étape (un admin satisfait toute étape) ; (3) **pas de double-signature** d'une étape par le même approbateur ; (4) la **barrière min-approbations** avant d'avancer ; dernière étape → `approved`, tout `reject` → `rejected` (les deux résolvent). L'application effective de la décision approuvée reste au demandeur via le `payload` (prochaine itération : appliquer automatiquement).
- **Délégations** : `delegation_usecases.go` — Create (refuse auto-délégation / perms vides / `end<=start`), List (projection `expired` en lecture si la fenêtre a expiré), Revoke, `ResolveEffectivePermissions` (union des droits des délégations actives *maintenant* pour un utilisateur — ce qu'une couche d'autz OR-erait à ses permissions propres).
- **Vérifier les permissions dans le code (middlewares)** : les routes sont montées sous `protected := api.Use(middleware.Protected(...))`. Gardes disponibles : `middleware.RequirePermission("risks:read")` (match wildcard : `risks:*` couvre `risks:read`, `*` couvre tout — lit `c.Locals("permissions")`) ; `middleware.RequireRole("admin", "root")` (lit `c.Locals("org_roles")`). Dans un handler : `middleware.GetContext(c)` → `{UserID, OrganizationID}` (helpers partagés `tenantID(c)`/`userID(c)`), et `middleware.GetUserClaims(c)` → `*authpkg.Claims` (`HasPermission("*")`, `OrgRoles`). Les routes governance : audit-trail + config workflow = `RequireRole("admin","root")` ; délégations + décisions d'approbation = membre authentifié (les use cases imposent four-eyes/rôle en interne). **Piège Fiber** : les sous-chemins statiques (`/governance/audit-events/export`, `/governance/delegations/effective`) sont montés **avant** les routes `/:id`.
- **Endpoints** : `/governance/audit-events` (+`/export` CSV, admin), `/governance/delegations` (GET/POST, `/:id/revoke`, `/effective`), `/governance/workflows` (CRUD admin), `/governance/approvals` (GET inbox + POST submit, `/:id` + `/:id/decide` + `/:id/cancel`). **Frontend** `features/governance` : `GovernancePage` à 4 onglets (`/governance`, item sidebar « Gouvernance » groupe Admin, i18n `n_governance`) — journal d'audit interactif (**diff Avant→Après** coloré par champ + filtres + export CSV), boîte de réception d'approbation (chaîne d'étapes + approuver/rejeter/annuler), délégations (create/revoke), constructeur de workflow (add/remove étapes). Service/hooks typés `governanceService.ts`/`useGovernance.ts` (zéro `any`).

## RBAC — Rôles métiers GRC (Business Roles)
Système de **rôles métiers** par-dessus le RBAC runtime existant. **Additionnel** — le chemin d'autorisation effectif (login → `OrganizationMember.Role` → JWT `permissions` → `middleware.RequirePermission`) est conservé ; on donne enfin à un membre `user` un moyen cohérent d'obtenir les chaînes de permission que les routes vérifient. Branche `feature/rbac-business-roles`. Détail complet : `docs/RBAC_BUSINESS_ROLES.md`.
- **Constat d'audit** : seuls 3 rôles d'org existent (`root`/`admin`/`user`) ; `root`/`admin`→`["*"]`, mais un `user` ne recevait ses permissions que d'un `Profile` (vocabulaire étroit, jamais aligné sur `risks:create`, `compliance:frameworks:read`, `vulnerabilities:read`, `scanner:*`, `automation:*`, `reports:board:*`, `incidents:*`) → **un `user` était verrouillé hors de la majorité de l'app**. Le frontend n'injectait même pas `user.permissions` (elles vivent dans le JWT) → `usePermissions()` niait tout. Deux systèmes RBAC hérités (`domain/rbac.go`+`RoleService`, `RequirePermissions` pluriel) pilotent des écrans d'admin mais **pas** l'autorisation runtime.
- **Source de vérité pure** : `internal/domain/business_roles.go` (stdlib, testable) — `PermissionCatalog` (46 permissions = **exactement** les chaînes des gardes de route, libellés FR/EN + groupe) et **11 présets least-privilege** : `rssi`, `dsi`, `risk_manager`, `auditor`, `compliance_officer`, `internal_control`, `asset_owner`, `risk_owner`, `security_analyst`, `executive`, `viewer` — chacun → liste de permissions + `default_landing`. `ValidateBusinessRoles()` (test) garantit qu'aucun préset ne référence une permission hors catalogue. Le **Tenant Admin** = rôle d'org `admin` (`*`), pas un préset.
- **Résolution** : `OrganizationMember.BusinessRole` (colonne, migration `0038`, AutoMigrate) + `OrganizationMember.EffectivePermissions()` — **point unique** : root/admin→`["*"]`, sinon union(préset, règles `Profile`) dédupliquée. Utilisé au **login** (`application/auth/login.go`, renvoie `business_role`) **ET** au refresh (`SessionResolver` dans `main.go`) → un rôle métier survit au refresh.
- **API** : `GET /rbac/business-roles` (catalogue+présets, tout membre), `GET /rbac/members` (membres+accès résolu, admin), `PUT /rbac/members/:userId/business-role` (affecter/effacer un rôle métier + option changer le rôle org ; `root` protégé). Use cases `application/rbac` + `GormMemberRBACRepository` (tenant-scoped, Update scopé id+org). **Incidents** convertis de `RequireRole("admin","analyst")` (où « analyst » n'est pas un rôle runtime → admin-only de fait) vers `incidents:*`.
- **⚠️ Bug corrigé (fuite de garde)** : `protectedMarketplace := protected.Use(RequireRole("admin","analyst"))` ajoutait la garde à **tout `protected`** (Fiber `Use` = toutes les routes suivantes) → automation/CTI/scanner/vulns/gouvernance/RBAC héritaient d'une garde admin-only, invisible car tout était testé en root. Scopé en garde par-route.
- **Frontend** : `lib/jwt.ts` (décode le JWT, sémantique wildcard identique au backend) → `useAuthStore` peuple `user.permissions`/`org_roles`/`business_role` au login+refresh ; **nav filtrée par permission** (`visibleNavGroups`, Sidebar + palette ⌘K) ; **landing par rôle** (`landingForBusinessRole`) ; écran **`/settings/roles`** (`features/rbac` : matrice de permissions + affectation inline par membre, admin) ; libellé de rôle dans la barre latérale. Zéro `any`.
- **Preuve live** (binaire :8090, Postgres:5434+Redis) : `user`+`rssi` → affectation API (23 perms résolues ; rôle inconnu **400**) → login (`business_role: rssi`, JWT 23 perms, pas de wildcard) → gardes réelles : `/risks`,`/automation/rules`,`/cti/vulnerabilities`,`/vulnerabilities`,`/compliance/frameworks`,`/rbac/business-roles`→**200** ; `POST /incidents`→passe la garde ; `DELETE /risks/:id`→**403** ; `/rbac/members`,`/marketplace/apps`,`/governance/audit-events`→**403** ; admin inchangé. `go build`/`vet`/`go test ./internal/domain/... ./internal/application/rbac/... ./internal/application/auth/...` verts, `tsc -b`/`vite build` verts. **Restes honnêtes** : scope `own`/`assigned` (Asset/Risk Owner) pas encore appliqué au niveau requête (accès tenant-large) ; rôles personnalisés par tenant via UI = prochaine itération ; gardes de route frontend = nav masquée + 403 backend (dégradation gracieuse), pas de `PermissionRoute` par écran.
